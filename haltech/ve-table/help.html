VE Table Tuning — Help

Analyzes Haltech NSP datalogs against your VE table to compute per-cell corrections.

Quick Start

  1. Export a datalog from NSP as CSV.
  2. In NSP, right-click your VE table → Copy Table → paste into a text file and save as .csv.
  3. Load both files, adjust filter settings, click Analyze VE.
  4. Review the correction map — positive = add fuel, negative = remove fuel.
  5. Download the corrected VE CSV and paste it back into NSP.

Input Files

NSP Datalog (.csv)

Required channels: RPM, Manifold Pressure or Fuel - Load (MAP), Wideband O2 Bank 1/2, Target Lambda.

Optional: O2 Control Short/Long Term Fuel Trim, Coolant Temperature, Throttle Position.

VE Table CSV

Exported from NSP using Copy Table. The tool expects the Haltech format with AxisValues1: (MAP in mbar), AxisValues2: (RPM), and CellValues: sections.

Filter Settings

Min Hit Count (default: 50)

Cells need more than this many samples to produce a correction. Higher = more confidence but needs more data.

Min Coolant Temp (default: 55°C)

Discards cold-engine samples where fueling is in warmup enrichment mode.

Min Run Time (default: 60s)

Skips the first N seconds of the log to avoid cold-start enrichment and sensor stabilization.

Min Change Amount (default: 0%)

Zeros out corrections smaller than this. Set to 1-2% to ignore noise on a final pass.

Scaling Factor (default: 100%)

Apply only a fraction of the correction. Use 75% for a conservative first pass, 100% for final tuning.

Tip: Start at 75% and iterate. Each pass should show smaller corrections as the table converges.

Outlier Filter (default: 0 = off)

Two-pass analysis: first computes per-cell mean and standard deviation, then discards samples beyond Nσ. Removes transient spikes from tip-in/tip-out.

Recommended: 2.0–3.0σ. Start with 2.5 if unsure.

Max TPS Rate (default: 0 = off)

Discards samples where throttle is changing faster than this rate (%/second). Ensures only steady-state data where lambda accurately reflects VE table contribution.

Recommended: 30–100 %/s. Lower = stricter steady-state. For track data try 50, for street driving try 100.

Smoothing (default: 0 = off)

Gaussian spatial smoothing applied to the correction grid. Blends each cell with its neighbors to reduce cell-to-cell noise.

Caution: Over-smoothing can mask real VE transitions (cam changeover, resonance effects). Keep at 1-2 unless data is very noisy.

Display Views

Correction %

Primary output. Green = add fuel (lean), Red = remove fuel (rich), Grey = insufficient data.

New VE Values

Complete corrected table. Green-highlighted cells were modified; grey cells retained original values.

Hit Counts

Sample density per cell. Identifies areas needing more driving time.

Std Dev

Data quality indicator. Green (≤1) = tight data, Yellow (≤3) = moderate, Red (>3) = noisy. High stddev cells benefit from the outlier filter.

CL Trim

Shows the closed-loop fuel trim contribution per cell. Large trim values indicate the ECU is already compensating heavily — the VE table needs the most work in those areas.

Recommended Workflow

  1. Gather data: Drive through varied RPM/MAP conditions after warmup. 10+ minutes of mixed driving.
  2. First pass: Scaling 75%, Outlier 2.5σ, TPS rate 50, Min change 1%.
  3. Review Std Dev: Red cells = noisy data. Consider more driving or stricter TPS filter.
  4. Apply and re-log: Paste corrected table into NSP, drive again, re-analyze.
  5. Final pass: Scaling 100%, lower min change to 0. Corrections should be small (±1-2%).
  6. Done: When the correction map shows mostly zeros across your operating range.
Pro tip: Save your VE_Difference.csv from each iteration to track convergence.
