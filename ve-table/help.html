VE Table Tuning Tool — Help

This tool analyzes MoTeC log data against your current VE table to compute per-cell corrections that bring your actual air/fuel ratio in line with your targets.

Quick Start

  1. Export a log from MoTeC i2 as CSV, or use the raw .ld file directly.
  2. Export your current VE table from MoTeC M1 Tune as a CSV (M1 v2 format).
  3. Load both files into the tool and click Process.
  4. Review the correction table — positive values mean "add fuel," negative means "remove fuel."
  5. Download the VE_New_Values.csv and import it back into M1 Tune.

Input Files

MoTeC Log File (.csv or .ld)

The tool accepts either a CSV export from MoTeC i2 or a raw .ld binary log file. Required channels:

  • Engine Speed — RPM for binning into the VE table columns
  • Inlet Manifold Pressure — MAP (kPa) for binning into VE table rows
  • Exhaust Lambda Bank 1 / Bank 2 — actual measured lambda (or "Exhaust Lambda" as fallback)
  • Fuel Mixture Aim — target lambda the ECU is commanding

Optional channels (used if present):

  • Fuel Closed Loop Control Bank 1/2 Trim — closed-loop trim already applied by the ECU
  • Coolant Temperature — for the coolant temperature filter
  • Throttle Position — for the TPS rate-of-change filter

VE Table CSV (M1 v2 format)

This is the CSV exported from MoTeC M1 Tune. The tool expects the standard M1 v2 header format:

M1,v2

Body,%

Cols,rpm,Eng Speed

Rows,kPa,Inlet Man Pres

,1000,1500,2000,... (RPM breakpoints)

30,45.2,46.1,... (MAP breakpoint + VE values)

Filter Settings

Min Hit Count (default: 50)

A cell must have more than this many valid samples to produce a correction. Cells below this threshold are greyed out and left unchanged in the output. Higher values give more statistical confidence but require more log data.

Tip: For a first pass with limited data, try 20–30. For a final tune, use 50–100+.

Min Coolant Temp (default: 55°C)

Discards all samples where coolant temperature is below this value. This ensures you're only analyzing data from a fully warmed-up engine where fueling is in its normal operating mode.

Min Run Time (default: 60s)

Discards all samples from the first N seconds of the log. This filters out cold-start enrichment and initial sensor stabilization.

Min Change Amount (default: 0 = disabled)

If a cell's average correction is smaller than this absolute value, it gets zeroed out. Useful for ignoring noise — if you set this to 1.0, any cell recommending less than ±1% change will be left alone.

Example: Set to 2.0 — a cell with a correction of +1.5% becomes 0 (no change). A cell with +3.2% is kept as-is.

How Corrections Work

For each valid log sample, the tool computes:

correction = ((lambda_target / lambda_actual) - 1) × 100 + CL_trim_avg

  • Positive correction → engine is running lean → VE table needs to be increased (add fuel)
  • Negative correction → engine is running rich → VE table needs to be decreased (remove fuel)

Each sample is binned to the nearest RPM/MAP cell in your VE table. The final correction for each cell is the average of all samples that landed in it.

Important: The closed-loop trim (CL_trim) is included in the calculation. This accounts for corrections the ECU has already made, giving you the total error including what closed-loop is compensating for.

Scaling Factor

Default: 100% (apply the full recommended correction).

Set this lower to apply only a fraction of the correction. This is a common conservative tuning practice — apply 50–75% per iteration to avoid overshooting, especially on a first pass or when you're unsure about data quality.

Example: A cell shows +6% correction. With scaling at 75%, the applied correction becomes +4.5%.

Tip: Use 100% when you have high-quality data (lots of hits, low std dev, steady-state driving). Use 50–75% on early iterations or when the std dev is high.

Outlier Filtering

Default: 0 (disabled).

When enabled, the tool performs a two-pass analysis:

  1. First pass: Bins all samples and computes the mean and standard deviation for each cell.
  2. Second pass: Discards any sample whose correction is more than Nσ (standard deviations) from its cell's mean, then recomputes the final averages.

This removes transient spikes (tip-in/tip-out, sensor glitches, momentary misfires) that would otherwise skew the average.

Example: Set to 2.0 — a cell with mean correction +3% and std dev 1.5% will discard any sample with correction outside the range [-0%, +6%].

Recommended values: 2.0–3.0σ. Lower values are more aggressive (discard more data). Start with 2.5 if unsure.

TPS Rate Filter

Default: 0 (disabled).

Discards samples where the throttle position is changing faster than the specified rate (%/second). During rapid throttle changes, lambda readings lag reality due to sensor response time and fuel transport delay. Filtering these out gives you only steady-state data where the lambda reading accurately reflects the VE table's contribution.

Example: Set to 50 — any sample where TPS is changing faster than 50%/second is discarded.

Recommended values: 30–100 %/s depending on how aggressive your driving style is. Lower values are stricter (more steady-state only). For track data with lots of transients, try 50. For street driving, 100 may be sufficient.
Note: This filter requires the "Throttle Position" channel in your log. If it's not present, the filter is automatically disabled and a warning is shown.

Display Views

After processing, four table views are available via the toggle buttons:

Correction %

The primary output. Shows how much to adjust each VE cell. Green = add fuel, red = remove fuel, grey = insufficient data.

New VE Values

Shows the complete recommended VE table with corrections already applied. Cells that changed are highlighted; unchanged cells are greyed out. Useful for a quick visual sanity check that the resulting table looks smooth.

Hit Counts

Shows how many valid samples landed in each cell. Green cells exceeded the hit threshold and contributed to corrections. This helps you identify areas of the map with insufficient data coverage — you may need more driving time at those RPM/MAP combinations.

Std Dev

Shows the standard deviation of corrections within each cell. This indicates data quality and confidence:

  • Green (≤1.0) — tight clustering, high confidence in the correction
  • Yellow (≤3.0) — moderate spread, correction is reasonable but not perfect
  • Red (>3.0) — high variance, treat this correction with skepticism. Consider enabling the outlier filter or gathering more data.

3D Surface Plot

A 3D wireframe surface visualization appears below the table. It renders the selected data (corrections, new VE values, or std dev) as a color-coded surface over the RPM × MAP grid.

Controls

  • Drag — rotate the view (left-right = yaw, up-down = pitch)
  • Scroll wheel — zoom in/out
  • Data dropdown — switch between Correction %, New VE Values, or Std Dev
  • Reset View — return to the default viewing angle
Tip: The 3D view is especially useful for spotting discontinuities or spikes in the correction map that might indicate bad data in a specific cell. A smooth surface suggests consistent, trustworthy corrections.

Copy to Clipboard

The "Copy Table to Clipboard" button copies whichever table view is currently displayed as tab-separated values. This pastes cleanly into:

  • Microsoft Excel
  • Google Sheets
  • LibreOffice Calc
  • Any text editor (as a tab-delimited table)

The button briefly shows "Copied!" as confirmation. Switch between views (Correction %, New VE Values, etc.) and copy whichever one you need.

Output Files

VE_Difference.csv

A simple grid showing the correction percentage for each cell. Cells below the hit threshold are empty. This is useful for record-keeping or importing into a spreadsheet for further analysis.

VE_New_Values.csv

A complete M1 v2 format VE table with corrections applied. This file can be imported directly back into MoTeC M1 Tune to replace your current VE table. Cells that didn't meet the hit threshold retain their original values unchanged.

Recommended Workflow

  1. Gather data: Drive the car through as many RPM/MAP combinations as possible. Steady-state cruising, part-throttle acceleration, and highway driving all contribute. Aim for at least 5–10 minutes of varied driving after warmup.
  2. First pass (conservative):
    • Set scaling factor to 75%
    • Set outlier filter to 2.5σ
    • Set TPS rate to 50 %/s
    • Set min change to 1.0%
  3. Review results: Check the Std Dev view. If many cells show red (>3σ), your data may have too many transients — try lowering the TPS rate filter or gathering more steady-state data.
  4. Apply and re-log: Import the new VE table, drive again, and re-analyze. Each iteration should show smaller corrections as the table converges.
  5. Final pass (full correction): Once corrections are consistently small (±1–2%), switch to 100% scaling and lower the min change threshold to capture the remaining fine adjustments.
  6. Convergence: You're done when the correction table shows mostly zeros or very small values across the operating range you care about.
Pro tip: Keep your VE_Difference.csv files from each iteration. Comparing them shows how quickly the table is converging and whether any cells are oscillating (which might indicate a sensor issue or a cell that spans too wide an operating range).
